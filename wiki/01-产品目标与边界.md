# 产品目标与边界

## 1. 产品目标

Daily Intelligence 要把“每天看很多站点”变成一份结构稳定、证据可追溯、早晚连续的中文日报。最终读者不需要理解抓取状态、JSON ID 或评分公式，只需看到：

- 来源内经过相对重要性排序的新闻列表；
- 每条信息的原文标题、自然中文翻译、中文 TL;DR 和原文链接；
- 来源原始热度/榜单位置；
- 基于当日主要事件形成的三个独立专业视角；
- 晚间版相对晨间版的新增、确认、修正和次日观察项；
- 本地交付后独立评估与可编辑用户反馈。

固定结构是产品契约，不由每日来源内容决定：

| 一级部分 | 二级栏目 | 空栏目行为 |
| --- | --- | --- |
| 资讯 | 国际、国内新闻、军事、市场 | 保留栏目并写覆盖说明 |
| 技术 | 技术新闻、值得阅读的论文、今日值得关注的开源项目 | 保留栏目并写覆盖说明 |
| 研判 | 地缘政治、AI 研究/开发、股票分析 | 三个 domain 必须分开 |

## 2. 时间契约

默认时区为 `Asia/Shanghai`。

| 版本 | 采集窗口 | 历史优先级 | 目标 |
| --- | --- | --- | --- |
| 晨报 | 前日 18:00 至当日 06:00 | 最近晚报 | 建立当日基线，指出需要继续观察的事项 |
| 晚报 | 当日 06:00 至 18:00 | 当日晨报 | 补充新增，确认或推翻晨间判断，形成次日观察项 |

采集窗口是编辑和时效性判断的边界，不意味着只有窗口内发布的内容才能出现。未在近期日报出现过的旧闻仍可作为背景或遗漏补充，但不得标成 `NEW`。Python 会在可解析发布时间不属于今天/昨天时把 `NEW` 降级为 `WATCH`。

## 3. 四层语义模型

```text
ArticleItem：某个来源页面上观察到的一条链接
Brief：日报里按来源展示的一条轻量信息
Event：支撑研判、需要证据链和连续 ID 的精选现实事件
Analysis：基于一个或多个 Event 形成、可更新或推翻的判断
```

### ArticleItem

是采集层事实。它只说明“在某个来源观察到某个标题和链接”，可能只有标题，也可能有公开 description、发布时间或按需正文。Python 不在这里判断新闻是否重要。

### Brief

是覆盖层和排版单位。它必须引用一个真实 `item_id`，由 Hermes 编写翻译、TL;DR、相对重要性和状态。每个成功来源有 `report_target`，有足够真实候选时必须达到目标；不存在“低于 60 分就不展示”的阈值。

### Event

是研判和连续性单位。schema 1.5 每版最多 12 个精选事件，通常建议 6—10 个。Event 需要稳定 ID、来源引用、置信度、证据说明、重要性拆分和状态。一个 Event 可以引用一个或多个 ArticleItem；当前写作流程倾向先建立单来源精选事件，模型也可在证据充分时合并多来源。

### Analysis

是可以被后续证据支持、修正、关闭或推翻的观点。必须引用 Event，而不能只写“据新闻可见”。三个领域固定为：

- `geopolitics`：地缘政治专家视角，同时覆盖中国与西方立场；
- `ai_technology`：AI 研究/开发工程师视角，给出学习或研究建议；
- `markets`：股票分析师视角，讨论趋势和风险，但不自动执行交易。

## 4. 为什么不把所有新闻都做成 Event

假设 28 个来源每个展示 8 条，覆盖层可能有 224 条。若每条都生成完整事实链、反证、情景和长期状态：

- 模型时间和 token 随新闻数量线性增长；
- 大量一般新闻挤压真正重要事件；
- Schema 更复杂，验证失败概率上升；
- 历史状态库被低价值事件污染。

因此系统用 briefs 提高信息密度，用少量 events 支撑深度研判。这也是“新闻可以多、正文不必全读”的技术基础。

## 5. 质量目标

一份可交付日报至少应满足：

1. 七个内容栏目全部存在；
2. 有真实候选的来源达到 `brief_plan.target_count`，且单来源不超过 15 条；
3. 外文标题保留原文并有实质中文翻译；
4. TL;DR 是中文事实摘要，不是“详见原文”“暂未获取摘要”等工作流提示；
5. 精选事件的 URL、标题、access 和发布时间与索引一致；
6. `NEW` 没有伪装旧闻；
7. 三类研判分开，且总计引用至少 60% 的精选事件；
8. 晚报明确写出日间变化和次日观察项；
9. JSON/Markdown 已保存为不可变 revision；
10. 发布后评估与报告 ID/hash 绑定，失败不撤回主报告。

## 6. 明确不做的事

- 不保证每个网站每次都可访问；系统保证诚实记录失败并继续。
- 不破解验证码、不绕过付费墙、不伪造浏览器指纹、不用代理规避限流。
- 不根据未读取正文补写细节。
- 不允许 Python 用字符串模板替代 Hermes 生成翻译和摘要。
- 不让生成 Agent 给自己的日报打分。
- 不把 Notion 作为唯一存储。
- 不直接交易，也不把一般市场分析包装成个性化仓位建议。

## 7. 边界带来的收益

| 边界 | 直接收益 |
| --- | --- |
| Article 不等于 Event | 允许多个来源描述同一现实变化 |
| Brief 不等于 Event | 扩大覆盖而不扩大重型分析成本 |
| 语义归 Hermes、机制归 Python | 降低提示词漂移和硬编码自然语言 |
| 本地先于远程 | Notion 故障可重试，不丢日报 |
| 生成与评估隔离 | 缩短发布等待，减少自评偏差 |
| 失败不等于无内容 | 读者能区分来源沉默与访问失败 |
| 早晚版共用连续状态 | 判断可以更新、反驳和关闭，而非每天重置 |
