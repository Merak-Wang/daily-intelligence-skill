# 测试、运维与演进

## 1. 当前测试基线

v1.0.0 的准确用例数由 `pytest --collect-only` 与 CI 给出，不把易变化的总数写成质量指标：

| 文件 | 主要覆盖 |
| --- | --- |
| `tests/test_reporting.py` | Schema、编译、中文/TL;DR、证据绑定、排序、时效、分析关系 |
| `tests/test_architecture.py` | workflow、状态、正文并发、自动验证、Notion、评估、不可变 artifact |
| `tests/test_prefetch.py` | 无脚本 HTML、来源预取并发、Edge 回退和限流停止 |
| `tests/test_semantics.py` | 内容指纹、评估门槛、context/编译复用 |
| `tests/test_normalize.py` | URL、来源过滤、adapter 规范化 |
| `tests/test_config.py` | 路径、唯一数据根、动态栏目页、配置边界 |
| `tests/test_importer.py` | 旧 JSON 导入与状态映射 |
| `tests/skills/test_daily_intelligence_skill.py` | Hermes skill 元数据与触发约束 |

测试数量不是产品质量分数，只是当前回归基线。新增行为应增加断言，不应为了保持数字不变合并掉关键用例。

## 2. 测试层次

### 纯函数单元测试

适合 canonical URL、item ID、filter、status classifier、batch balancing、importance allocation 和 taxonomy。输入固定，不启动浏览器或网络。

### Artifact/状态测试

使用 `tmp_path` 验证 revision、latest、run 迁移、不可变写、连续状态和 legacy nested view 同步。

### Publisher 测试

mock `httpx` 或构造 `NotionPublisher`，验证 schema profile、属性类型、100-block 分批、registry 恢复和评估追加。测试不能使用真实 token。

### Skill 合约测试

检查 `SKILL.md` frontmatter、平台路径和关键流程要求，防止仓库代码已更新但 Hermes 仍读旧说明。

### 真实环境冒烟

发布前人工运行少量来源、Edge profile、验证队列和可选 Notion sandbox。网络站点易变化，真实冒烟不应替代离线测试。

## 4. 每日运维检查

### 晨报/晚报完成后

- run 是 `completed` 或 `completed_partial`；
- `artifacts.index_path/context_path/json_path/markdown_path` 均存在；
- `pending_sources` 与页面状态一致；
- 报告没有 placeholder TL;DR、英文占位翻译或内部 access 文案；
- 三个 analysis domain 分开；
- 晚报包含 changes 与 tomorrow watch；
- Notion registry 是 `complete`，或本地已完成且远端错误明确；
- evaluation scheduler 是 `scheduled` 或有可执行失败原因；
- 评估最终 append 到准确 report ID/hash。

### 运行超时

检查 run.metrics：

- `elapsed_seconds`：从 run created_at 到完成；
- `deadline_exceeded`：完成时是否超过预算；
- `selected_fulltext_items`：是否把正文硬上限用满；
- `attempt`：是否重复 restart。
- `phase_durations_seconds`：采集、context、正文、Agent 写作等待、验证/定稿和发布的真实分段时间。

同时检查 `artifacts.collection_metrics` 的 source/candidate 数、状态分布和预取并发配置。Agent 写作慢与采集慢应分别处理，不能再把整段 wall time 归因于网页抓取。

600 秒是流程预算，不是操作系统强制终止。若采集阶段长期超时，应优化来源等待/过滤或关闭低价值来源，而不是仅调大 token。

## 5. 每周质量检查

- 来源候选是否被导航、评论、标签页污染；
- report_target 是否与真实产出匹配；
- 某个来源是否长期 `partial`/`verification_required`；
- 来源原始 rank 是否保留；
- 旧闻是否误标 `NEW`；
- TL;DR 是否忠于 description/正文；
- metadata-only 精选事件是否过多；
- 分析是否引用大部分精选事件，而非泛泛议论；
- 独立评估是否长期给同一维度低分；
- 用户反馈是否被下一次 context 读取；
- 动态 source pages 是否仍有价值，应否移除。

## 6. 常见故障定位

| 现象 | 先看 | 常见原因 | 处理 |
| --- | --- | --- | --- |
| Hermes 找不到 skill | `hermes skills list`、安装目录 | 复制到错误的 `~/.hermes` | Windows 使用 `%LOCALAPPDATA%\hermes\skills\research\...` |
| 两套历史 | `data-root status`、run 的 `data_root` | Cron 与手动命令 DATA_DIR 不同 | 修正配置；确认迁移时才 `data-root adopt` |
| 评估不自动运行 | run.evaluation.scheduler、Gateway | Gateway 未常驻或 Hermes CLI 失败 | 启动 Gateway，再重试 finalize --publish |
| 每站新闻很少 | index items、source policy | 过滤过严、栏目页少、report target 低 | 先看 index，再改 pattern/栏目/target |
| `coverage source` 错误 | context.brief_plan | Hermes 未完成全部 batch 或丢 brief | 按 target_count 补齐真实 item ID |
| 外文标题无中文 | draft brief.title_zh | 子 Agent 返回不完整或用了占位前缀 | 由 Hermes 模型重写，不用 Python 前缀 |
| TL;DR boilerplate | validator 具体位置 | 使用“详见原文/暂未读取”模板 | 按证据层级写事实摘要 |
| finalize 卡在 finalizing | run.error | 旧版本状态未回退或进程中断 | 当前代码验证失败会回 awaiting_authoring；中断时重跑 finalize |
| Notion 400 schema mismatch | `configs/notion.yaml` 与真实 properties | Name/Title、status/select 不一致 | 修改映射，不自动改共享 data source |
| Edge 页面可打开但未采集 | 验证页顶部连接状态 | 直接双击静态 HTML、CLI 已退出 | 必须由交互式 `run-edition` 或 `verify-pending` 启动并保持 CLI 运行 |
| Reuters 暂时限制 | page status `rate_limited` | 站点限流 | 停止本轮重试，保留链接稍后人工查看 |

## 8. 演进方向

### 高优先级

- 用真实晨报/晚报持续校准 HTTP 预取命中率、Edge 回退率和 10 分钟预算；
- 加强 RSS、官方 API、论文与 GitHub adapter，减少脆弱 DOM 依赖；
- 增加 stale-lock 安全检测，而不是自动删除；
- 对来源探索做命中率、噪音率和耗时回收；
- 增加真实报告 fixture 和离线质量 eval。

### 中优先级

- 事件匹配辅助，减少 Hermes 手工聚类成本；
- 预测校准与后验复盘；
- 更细的正文证据片段定位；
- 多 publisher 投影；
- 独立投资 suitability 流程。

### 技术债说明

`beautifulsoup4` 当前在 `pyproject.toml` 声明，但主运行路径没有直接 import。删除前应确认打包、旧代码和下游使用；若确认无用途，应在独立依赖清理变更中移除并跑完整测试，不应顺手夹在来源功能修改中。

演进原则：先提高数据模型、证据和可恢复性，再扩大来源数量；先用配置扩展，再新增 adapter；最后才增加核心 workflow 分支。
