# 核心算法与跨模块调用

本页沿数据流解释关键算法。每节先说明实现，再说明为什么这样做和修改时会影响哪些模块。

## 1. URL 规范化与稳定 Article ID

实现：`utils.canonicalize_url()` 与 `utils.item_id()`。

规范化步骤：

1. 去掉首尾空白；
2. HTTP/HTTPS 统一为 HTTPS；
3. host 转小写并去掉开头 `www.`；
4. path 中多个 `/` 合并；
5. 非根路径删除末尾 `/`；
6. 查询参数删除以 `utm_`、`guce_`、`guccounter`、`ref`、`source` 开头的跟踪项；
7. 删除 fragment；
8. 保留剩余查询参数并重新编码。

ID：

```text
digest = SHA256(source_id + "|" + canonical_url)[:12]
item_id = source_id + "-" + digest
```

同一 URL 在不同来源下可能语义角色不同，因此 source ID 参与 hash。修改 canonicalization 会改变未来 item ID，并影响历史去重、Event 复用和旧报告识别；必须增加迁移/兼容测试。

## 2. 通用链接过滤

实现：`adapters.is_eligible()`。

一条链接必须依次通过：

```text
标题至少 8 字符、URL 为 HTTP(S)
→ 标题不是 low-information 文案
→ 标题含英文或 CJK 字符
→ 不命中 exclude_title_patterns
→ host 在 include_domains（若配置）
→ URL 不命中 exclude_patterns
→ URL 命中至少一个 article_patterns（若配置）
```

为什么 include 与 article pattern 同时需要：include domain 防止跳到外站，article pattern 防止同站导航、作者、标签、隐私页进入。

`_clear_misattributed_descriptions()` 还会检查至少 60 字符的 description：若完全相同的长描述出现在多个不同链接上，清空这些描述，避免卡片 DOM 选择错误把一个摘要复制给多篇文章。

修改影响：sources.yaml、adapter fixture、`test_normalize.py`，以及 context 中可用 TL;DR 证据。

## 3. 页面挑战与来源状态

实现：`collector.detect_challenge()` 与 `classify_source_status()`。

挑战信号：

- HTTP 401、403、429；
- 标题或 body 前 30000 字符中的 human/robot/security/captcha 等文本；
- captcha/challenge iframe；
- temporarily limited/restricted、too many requests 等限流文本。

优先级：

```text
rate_limited
> verification_required
> HTTP >= 400 的 failed
> 有 items 的 success
> no_items
```

检测到挑战时不调用 adapter，因为挑战页里的导航链接不是新闻。429 与明确限流单独分类，使人工队列和重试策略可以停止刷新。

## 4. 多栏目 round-robin 合并

实现：`collector.collect_source()`。

一个来源的 URL 顺序为主 URL、静态 explore URL、动态 approved URL。每页独立采集后，不是简单拼接，而是：

```text
page1[0], page2[0], page3[0],
page1[1], page2[1], page3[1], ...
```

同时按 canonical URL 去重，达到 source.max_items 即停止。最后根据合并顺序写 `metadata.source_rank = 1..N`。

这样 BBC 首页不会先吃满 80 条，导致 Business、Science、UK 页面永远没有候选。代价是 `source_rank` 表示合并后的来源顺序，不完全等于每个栏目各自榜单。`hot_rank/list_position` 会保留在 metadata 中，但当前 compiler 只在缺少 `source_rank` 时把它们作为 fallback。

## 5. Context 候选压缩

实现：`context._compact_candidates()`。

### 5.1 分组

根级 index items 按 `source_id` 分组。每条来源原始排名按以下优先级：

```text
metadata.source_rank
→ metadata.hot_rank
→ metadata.list_position
→ index 中出现顺序
```

当前代码实际先取 `source_rank`，因此 Weibo 若同时有 source_rank/hot_rank，压缩排序使用 source_rank；publisher 的 label 仍保留生成后的 rank。若要让热搜真实名次优先，应修改这一优先级并增加测试，不能只改文档。

### 5.2 候选排序

排序键按以下优先级：

1. 已有 `full_text/partial` 的 item；
2. 有可解析发布时间的 item；
3. 发布时间越新越前；
4. source rank 越小越前。

这不是最终重要性排序，只是决定哪些候选进入有限 context。最终展示顺序由 Hermes importance 决定，并以 source rank 打破同分。

### 5.3 数量限制

```text
base_limit = min(
  context_items_per_source,
  max(5, report_target * 2)
)
limit = max(base_limit, enriched_count)
```

默认 context 每来源最多 25，但若已 enrich 的候选超过 base limit，会全部保留，避免正文抓完却被 context 截掉。

Context 条目还写入 `source_candidate_rank`、`source_rank` 和 `previously_reported`。后者只检查最近最多五份连续性报告中的 brief item IDs。

## 6. 三批负载均衡

实现：`context._balanced_source_batches()`。

算法是按候选数降序的贪心装箱：

1. 统计每来源 candidate count；
2. 按 count 降序、source ID 升序排列；
3. 每次把来源放进当前总 candidate count 最小的 bin；
4. 最多创建 3 个非空 batch。

这是 Longest Processing Time 的简化形式，复杂度主要是来源排序 `O(S log S)`。它保证同一来源不会被拆到多个 worker，减少重复和合并冲突；同时比按来源数量平均更接近工作量均衡。

Python 只生成计划，实际并发由 Hermes 一次 `delegate_task` 完成。若 Hermes 逐批串行调用，算法仍正确但没有时间收益。

## 7. Brief Plan 目标计算

实现：`context._build_brief_plan()`。

每来源：

```text
target_count = min(
  当前 compact candidates 数,
  source.report_target,
  source.report_max,
  15
)
```

`default_item_ids` 取 compact 排序前 target_count 条。成功来源有足够候选时，validator 要求最终 briefs 达到这个 target；compiler 不再自动补 brief。

为什么不自动补：Python 无法生成自然中文翻译和有信息量 TL;DR。旧版自动补齐造成大量 boilerplate，后来改为 Hermes 并行完成全部语义。

## 8. 正文目标选择和并发

### 8.1 选择

`content._ordered_targets()`：

- 以根级 index items 建 item ID 映射；
- `dict.fromkeys(selected_ids)` 保序去重；
- 忽略未知 ID；
- 按传入顺序截到 effective limit。

`workflow.enrich_edition()` 还会去掉 previous selected IDs，并计算：

```text
remaining = hard_cap - previous_count
accepted = requested[:min(requested_limit, remaining)]
```

deadline 已过时 accepted 为空，并写 `budget_exhausted=true`。

### 8.2 并发

`content._run_parallel_extraction()` 为每个 domain 建 semaphore：

```python
async with domain_semaphore, global_semaphore:
    await _extract_one(...)
```

默认全局 3、同域 1。先等待 domain slot，再占 global slot，避免三个同域任务占住全局槽却都在等待同一域许可，从而饿死其他域。

正文任务通过 `asyncio.gather()` 并行；来源索引采集不是这套并发，目前仍顺序执行。

### 8.3 正文状态判定

页面通过挑战和 HTTP 检查后：

1. 删除噪音节点；
2. 从 og/twitter meta 更新标题、description、发布时间、图片；
3. 按 content selectors 找第一个至少 500 字符区域；
4. ≥1500 字符为 full_text；500—1499 为 partial；不足为 metadata_only；
5. 有 body 时保存 frontmatter + 正文 Markdown；
6. 失败写 content_error，不抛出阻塞其他并发任务。

## 9. 草稿编译

实现：`reporting.compile_report_data()`。这是降低 schema 失败的核心。

### 9.1 Envelope

编译器确定：schema 1.5、date、edition、language、generated_at、evaluation_status、changes/watch/pending defaults。缺 title 时生成确定性标题；executive_summary 字符串可规范成数组。

缺失 executive summary 时会使用中性的 fallback。这是少数允许 Python 生成的非证据语义，因为它不陈述外部事实。

### 9.2 Section 归一化

接受 list/mapping 草稿和已知 alias，例如 `information.economy → information.market`。Brief 的实际 section 由 index item 的 module/category 决定：放错时自动移动并给 warning；未知 item ID 的 brief 被删除并警告，绝不凭草稿 URL 创建新 item。

七个 section 按固定顺序创建，即使为空。

### 9.3 标题与翻译

`_normalize_brief_title()`：

- `brief.title` 最终强制恢复为 index 原文标题；
- 删除 `[英]`、`[EN]` 等语言 marker；
- 外文标题需要 `title_zh` 含实质 CJK；
- 若 Agent 把中文译文误放 title，可迁移到 title_zh；
- 原文是中文时删除多余 title_zh。

编译器不会用 `【外文】`、`详见原文` 等前缀凑中文字符。

### 9.4 权威引用

`authoritative_ref(item_id)` 从 index 生成：title、url、access、role、published_at。Access 映射由 Python完成，避免 Agent 把 `not_fetched` 错写成不存在的 report 枚举。

### 9.5 来源排名

编译器计算 item rank，并生成：

- Weibo：`热搜TopN`；
- Hacker News、Lobsters、GitHub Trending：`榜单TopN`；
- 其他：`来源TopN`。

Brief 展示排序：importance 降序；同分时 source rank 小者优先。数值 importance 不显示给读者。

### 9.6 时效性降级

可解析 published_at 后，若 brief/event 标为 `NEW`，但相对报告日期不是 0 或 1 天，自动改为 `WATCH`。Event 没有可解析日期时按 8 天处理，用于 freshness cap；validator 对 schema 1.5 NEW 无日期仍给错误。

### 9.7 Event ID、分数与置信度

Event 缺 ID 且有权威 ref 时：

```text
EVT-YYYYMMDD-SHA256(first_item_id)[:8].upper()
```

Importance 总分限制到合法范围，再按各维度 cap 比例分配整数，余数按小数部分和 cap 排序逐一补齐。freshness cap 随最新证据年龄变化：

| 年龄 | freshness 上限 |
| --- | ---: |
| 0 天 | 15 |
| 1 天 | 13 |
| 2—3 天 | 8 |
| 4—7 天 | 4 |
| 8 天以上 | 1 |

若所有 source refs 都是 metadata_only 或 verification_required，confidence 上限为 0.65，并在精选事件 evidence_notes 中加入“未读取正文”披露。此披露不会作为 brief TL;DR 展示。

### 9.8 Analysis 引用转换

模型可在 `evidence_item_ids` 中直接给 item ID。编译器先建立 `item_id → event_id` 映射，再转换为去重后的 `evidence_event_ids`。不属于精选 event 的 item ID 被忽略并给 warning。

三个 domain 按 geopolitics、ai_technology、markets 排序，自动合并各 domain 的 required perspectives 和 assessment types。模型仍必须提供实质 facts/reasoning/narrative 等语义。

## 10. 报告验证

验证有三层：

### 10.1 JSON Schema

Draft 2020-12 检查 required、类型、枚举、范围、URL/date format 和对象结构。

### 10.2 证据身份

对 index 逐项检查：

- item ID 存在；
- title 和 canonical URL 与 index 一致；
- access 与 content status 映射一致；
- source/module/category 合法；
- published_at 和来源身份不冲突。

### 10.3 跨字段语义约束

包括但不限于：

- 七 section 完整；
- 有候选来源达到 report target；
- 单来源不超过 15 briefs；
- 外文 title_zh 有实质中文；
- TL;DR 非空、有至少四个 CJK 字符、不等于标题、不命中 boilerplate；
- 精选 event ≤12，且有 brief featured link；
- event_count/brief_count/source metrics 正确；
- NEW 日期与历史复用合法；
- metadata-only evidence 有披露；
- 三个 analysis domain 都存在；
- trend/risk/learning-research 类型覆盖；
- 中国/西方/AI/股票等视角覆盖；
- analyses 引用至少 60% 精选 events；
- 晚报有 changes 和 tomorrow watch；
- schema 1.5 evaluation_status=pending 且不含内嵌评估。

Warnings 记录可交付但需关注的问题，例如同来源多个 brief importance 完全相同。Errors 阻止保存和发布。

## 11. 保存、hash 与连续状态

`reports.save_report()` 顺序：

```text
deepcopy draft
→ 对齐 index date/edition
→ 分配 report revision 与 ID
→ compile
→ normalize counters
→ validate
→ write immutable JSON
→ render/write Markdown
→ update latest
→ 旧 schema 或含评估报告才立即更新 state
→ 计算 content hash
```

Hash 对完整 report dict 做稳定 JSON 序列化后 SHA-256，用于评估身份。

schema 1.5 的主报告不立即更新连续状态；`save_evaluation()` 把评估附到 report 副本后再调用 `update_continuity_state()`，使低质量内容可以被排除。

## 12. 连续性隔离算法

`context._load_reports()` 读取全部 report，按 generated_at 降序：

- 晚报先选同日晨报；
- 晨报先选更早日期的晚报；
- 再补其他未重复 report；
- 最多取 5 份。

每份报告寻找对应独立 evaluation。没有评估时默认 `selective`，排除 formatting、event_summaries 和 analyses，只保留最低限度身份，防止旧错误扩散。

`evaluation_continuity_floor()`：

- 总分 ≤22 或四个关键维度中至少三个 ≤2：强制 reject all；
- summary accuracy ≤2：至少排除 event summaries；
- factual reliability ≤2：至少排除 source access；
- total <32 或关键维度低分：accept 被降为 selective；
- selective 没写排除项：至少排除 event summaries。

这是评估 Agent 决策之上的确定性污染防护。

## 13. Notion 发布算法

### 13.1 Schema profile

依次尝试 YAML profiles，第一套属性名和类型全部匹配的 profile 生效。

### 13.2 日期页面复用

用 date 属性查询同日报页面；存在则更新 properties，不存在则创建。晨报与晚报因此投影到同一日期页面，由 block 内容追加形成版本演进。

### 13.3 幂等和续传

registry key 为 `date:edition`。非 force 且已 complete 时返回 `skipped_duplicate`。publishing 中断时验证 report_id 一致，从 `blocks_appended` 继续每批 100 blocks。

### 13.4 评估追加

`append_evaluation()` 要求主报告已发布，并用 evaluation ID 去重。评估失败不改变主 registry complete 状态。

## 14. 人工验证前端调用链

```text
cli.run_pending_verification(index)
→ cli.pending_verification_pages(index)
→ write_verification_queue(local HTML)
→ Playwright persistent Edge context
→ wait_for_clicked_verifications
→ 用户点击打开新标签
→ wait_for_visible_verification
→ capture_verified_page
→ collector.collect_loaded_page
→ merge_verified_results
→ workflow.adopt_index_for_run
```

静态 HTML 只是 UI；采集依赖仍在运行的 CLI 与浏览器 context。直接双击 HTML 没有本地桥接，因此页面能打开但不会保存 JSON。

`run-edition --open-verification` 在 `prepare_edition()` 返回后调用该函数；手动 `verify-pending` 也调用同一函数，因此自动和手动流程不会形成两套采集/合并逻辑。没有 pending page 时函数在启动 Playwright 前返回 `no_pending_pages`。

成功的捕获不会覆盖原 index，而是创建新 revision。已发布 run 会保存 previous_report 元数据并回到 awaiting_selection，符合“本地历史不可覆盖、补充版本可追溯”的原则。

## 15. 修改算法时的跨模块影响表

| 修改 | 同步检查 |
| --- | --- |
| URL/ID | utils、importer、去重、历史连续性、normalize tests |
| 来源状态 | models、collector、CLI 验证队列、pending report、tests |
| source rank | collector、context、compiler、renderer、tests |
| report target | config validation、context plan、report validator、docs |
| 正文状态 | models、content、access mapping、confidence、tests |
| section/category | taxonomy、schema、compiler、renderers、configs、tests |
| analysis 字段 | schema、contract、compiler、validator、state、Notion、Markdown |
| evaluation 规则 | reporting floor、context history、state、tests |
| Notion mapping | notion config、schema validator、publisher tests、setup docs |

任何跨模块变化都应先写失败测试，再修改最底层真源，最后同步上层 renderer 和文档。
