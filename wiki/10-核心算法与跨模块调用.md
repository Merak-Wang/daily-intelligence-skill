# 核心算法与跨模块调用

## URL 规范化和条目 ID

`utils.canonicalize_url()`：

1. 将 HTTP(S) 统一为 HTTPS；
2. 主机名转小写并移除 `www.`；
3. 合并重复 `/`，移除非根路径尾部 `/`；
4. 移除 Fragment；
5. 移除 `utm_`、`guce_`、`guccounter`、`ref`、`source` 前缀的查询参数；
6. 保留其余查询参数。

`item_id()` 计算 `sha256(source_id + "|" + canonical_url)`，取前 12 位并加来源前缀。同一文章在不同来源下允许拥有不同 ID。

涉及模块：`utils.py`、`adapters.py`、`collector.py`、`reporting.py`。测试位于 `tests/test_normalize.py`。

## 链接筛选

`adapters.is_eligible()` 依次检查：

- 标题长度和字符信息量；
- HTTP(S) URL；
- 低信息标题黑名单；
- `exclude_title_patterns`；
- `include_domains`；
- `exclude_patterns`；
- `article_patterns`。

通用导航规则和来源专用规则都必须通过测试。放宽 `article_patterns` 时要增加作者页、标签页、评论页和站内搜索页的拒绝样例。

## HTTP 预取与浏览器回退

`prefetch_browser_pages()` 只处理 `browser_index` 来源：

1. 使用 `httpx` 并发请求；
2. 将响应体截断到 2 MB；
3. 使用 BeautifulSoup 提取链接和邻近时间文本；
4. 分类访问限制；
5. 调用通用行解析器生成候选。

`page_needs_browser()` 的主要规则：

- 已成功取得候选：不回退浏览器；
- 限流：不在同一轮回退浏览器；
- 401、403、空公开页面或专用 Adapter：允许浏览器处理；
- 其他确定的 HTTP 失败：不重复导航。

预取并发由 `collection_global_concurrency` 和 `collection_per_domain_concurrency` 控制。

## 来源状态合并

多栏目来源分别采集，`collector.py` 合并为一个 `SourceResult`：

- 去重基于规范 URL；
- 多页候选按栏目轮转合并，避免第一栏目耗尽上限；
- 任一页面成功但其他页面失败时保留 `partial`；
- 访问错误优先于 `no_items`；
- 每个页面结果保存在 `page_results`，用于诊断。

写入 Index 时同时保留根级 `items` 和兼容的嵌套来源条目。

## Context 候选压缩

`context._compact_candidates()` 按来源分组，排序键为：

1. 已有全文或部分正文；
2. 有可解析发布时间；
3. 发布时间较新；
4. 原来源排名。

每来源基础上限为 `min(context_items_per_source, max(5, report_target * 2))`。已 Enrich 条目不会因基础上限被截掉。

Context 只保留写作需要的字段，并加入：

- `source_candidate_rank`；
- `source_rank`；
- `previously_reported`；
- `semantic_fingerprint`。

## Brief 批次和计划

`_balanced_source_batches()` 按来源候选数降序，将来源放入当前条目最少的批次，最多三个批次。同一来源不会跨批次。

`_build_brief_plan()` 为每个来源计算：

```text
target_count = min(候选数, report_target, report_max, 15)
```

计划保存默认条目 ID、可复用 ID、需要新写的 ID、目标栏目和批次。编译器按该计划检查覆盖，不自行生成缺失摘要。

## 正文选择与并发

`content._ordered_targets()` 按调用方 ID 顺序选择存在于 Index 的条目，并应用全文硬上限。`_run_parallel_extraction()` 使用：

- 全局 Semaphore；
- 按规范主机名的同域 Semaphore。

正文提取移除脚本、样式、导航、页脚和常见噪音节点，再按 `content_selectors` 选择可见文本。结果保存为 Markdown，并生成新 Index Revision。

`workflow._validate_enrichment_lineage()` 在 Finalize 前确认当前 Index 仍包含所有已成功 Enrich 的条目和状态，防止使用旧 Index 丢失正文证据。

## 草稿编译

`reporting.compile_report_data()` 是模型草稿到持久报告的唯一编译入口。主要步骤：

1. 归一化顶层字段和栏目映射；
2. 将旧栏目别名转为 schema 1.5 ID；
3. 删除未知、重复或栏目错误的 Brief；
4. 依据 Index 注入标题、URL、来源、访问状态和时间；
5. 生成稳定事件 ID、来源引用、计数和来源指标；
6. 计算重要性拆分和时效上限；
7. 规范 Analysis 引用；
8. 从 Index 生成待验证来源；
9. 返回可审阅警告。

编译器不会：

- 翻译标题；
- 编写 TL;DR；
- 生成缺失 Brief；
- 补写 Analysis；
- 把未知 ID 映射到相似文章。

## 报告校验

`validate_report_data()` 分三层执行：

1. JSON Schema；
2. 报告内部跨字段约束；
3. 报告与 Index 的证据身份约束。

关键检查包括：

- 七个固定栏目和标题；
- 三个分析域；
- 中文用户可见文本；
- TL;DR 占位文本和访问免责声明；
- 来源 ID、标题、URL、访问状态和时间一致；
- Brief 覆盖目标与每来源上限；
- 精选事件数量和来源条目约束；
- Analysis 引用覆盖和具名来源绑定；
- `NEW` 与发布时间的时效关系；
- 重要性总分与拆分一致；
- 晚报 `changes`、`tomorrow_watch_items`；
- Evaluation 总分、Hash 和连续性决定。

错误阻止保存和发布；警告随编译结果返回。

## 保存和连续状态

`reports.save_report()`：

1. 再次编译和校验；
2. 分配 Report Revision；
3. 保存 JSON 和 Markdown；
4. 更新可变 `latest` 指针；
5. 更新连续状态；
6. 生成本地输出。

`state.update_continuity_state()` 维护事件、研判和观察项的当前视图，并保存不可变历史 Revision。

`semantics.semantic_fingerprint()` 绑定条目证据。缓存内容只有在指纹未变化、Evaluation 通过门槛且未排除摘要时才可复用。改写后的语义重新进入待评状态。

## 本地输出

`local_output.write_local_outputs()`：

- HTML 对文本转义，只生成安全 HTTP(S) 外链；
- 日报首页扫描本地报告和 Evaluation 重建；
- PDF 优先使用配置引擎，Edge 失败时可降级 ReportLab；
- Evaluation 完成后只刷新投影，不修改报告 JSON 或 Hash。

## Notion 发布

`notion.publish_report()`：

1. 读取 Token 和 Data Source ID；
2. 获取目标 schema 并选择兼容 Profile；
3. 校验报告；
4. 按日期查找或创建页面；
5. 使用本地登记恢复已追加块位置；
6. 更新属性并完成登记。

`append_evaluation()` 使用同一登记定位页面。Notion 内容不参与报告编译，也不作为连续状态事实源。

## 变更影响表

| 修改 | 同步检查 |
| --- | --- |
| URL 规范化 | ID 稳定性、去重、旧 Index 引用 |
| 来源过滤 | 配置、Adapter、Normalize 测试 |
| 状态枚举 | Collector、Importer、Verification、报告待处理区 |
| Context 排序 | Brief Plan、语义缓存、覆盖测试 |
| 报告字段 | Schema、模板、编译、校验、所有投影 |
| Analysis 规则 | 模板、引用校验、Evaluation、连续状态 |
| Artifact 路径 | Runtime guard、恢复、安装器、`.gitignore` |
| Publisher | schema 校验、登记、重试和失败恢复测试 |
