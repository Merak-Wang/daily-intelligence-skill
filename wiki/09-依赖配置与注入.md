# 依赖、配置与注入

这里的“注入”不是某个 DI 框架，而是指外部值如何在运行时进入系统：CLI 参数、环境变量、YAML、动态来源页、adapter 注册表、Agent context、报告索引证据和 Notion schema。理解这些入口，才能解释为什么同一代码在 Hermes Desktop、Cron 和手动 PowerShell 中会出现不同结果。

## 3. 资源目录解析

Python 包需要找到 `SKILL.md`、`configs/sources.yaml` 和 `schemas/report.schema.json`。`config.project_root()` 按顺序检查：

1. `DAILY_INTEL_SKILL_DIR`；
2. 当前导入模块的 `parents[2]`；
3. Hermes Home 下 `skills/research/daily-intelligence`；
4. 当前工作目录；
5. Hermes skills 下任意 `*/daily-intelligence`。

候选必须同时包含三个关键资源才有效，否则抛出列出全部检查路径的 `FileNotFoundError`。

### 稳定安装与 editable 安装的区别

- 稳定 Windows 安装先用 robocopy 镜像仓库到 Hermes skills，再从镜像安装 Python 包；模块 `parents[2]` 通常就是已安装 skill。
- `-Editable` 从开发仓库安装；模块路径指向开发仓库，因此修改代码立即生效。
- 若系统同时存在旧镜像和 editable 包，Python 导入位置优先决定资源目录。调试时先检查 `daily_intelligence.__file__`，不要只看 Hermes skill 文件夹。

需要强制资源目录时设置：

```powershell
$env:DAILY_INTEL_SKILL_DIR = "E:\ai_project\daily-intelligence-skill"
```

## 4. Hermes Home 注入

`resolve_hermes_home()`：

1. 若有 `HERMES_HOME`，使用该绝对路径；
2. Windows 且有 `LOCALAPPDATA`：`%LOCALAPPDATA%\hermes`；
3. 其他系统：`~/.hermes`。

因此 Windows 实际 skills 目录是：

```text
%LOCALAPPDATA%\hermes\skills\research\daily-intelligence
```

而不是 `~/.hermes/skills`。

## 5. `.env` 注入与进程边界

所有 CLI 子命令在解析参数后、加载 YAML 前执行：

```python
env_path = resolve_hermes_home() / ".env"
load_dotenv(env_path, override=False)
```

含义：

- `.env` 固定放 Hermes Home，不是仓库根目录；
- 当前 PowerShell/Cron 进程已有变量不会被 `.env` 覆盖；
- `daily-intel` 是独立进程，但它会自行加载 Hermes `.env`，不需要每次 `export $(grep...)`；
- 直接导入内部 Python 函数而不经过 `cli.main()` 时不会自动加载 `.env`，测试和嵌入式调用应显式传参或设置环境。

支持变量：

```text
HERMES_HOME
DAILY_INTEL_SKILL_DIR
DAILY_INTEL_DATA_DIR
DAILY_INTEL_PROFILE_DIR
DAILY_INTEL_BROWSER_CHANNEL
DAILY_INTEL_TIMEZONE        .env.example 中保留；当前 CLI 主要用 --timezone/YAML
NOTION_TOKEN
NOTION_DATA_SOURCE_ID
```

注意：`DAILY_INTEL_TIMEZONE` 当前未被 `load_config()` 直接读取；时区来源是 `--timezone` 或 sources.yaml。不要把示例变量误认为已实现的解析优先级。

## 6. CLI 参数注入与优先级

### 全局参数

```text
daily-intel [--config PATH] [--data-dir PATH] [--timezone ZONE] SUBCOMMAND ...
```

它们必须位于子命令前。解析规则：

| 值 | 优先级 |
| --- | --- |
| sources config | `--config` > `project_root()/configs/sources.yaml` |
| timezone | `--timezone` > sources.yaml `timezone` > `Asia/Shanghai` |
| DATA_DIR | `--data-dir` > `DAILY_INTEL_DATA_DIR` > `<Hermes Home>/daily-intelligence`；显式参数与环境变量冲突时失败 |
| profile | 子命令 `--profile-dir` > YAML 指定环境变量的值 > `<Hermes Home>/browser-profiles/daily-intelligence` |
| browser channel | 子命令 `--browser-channel` > YAML 指定环境变量的值 > YAML default > Windows `msedge` / 其他平台 Playwright 默认 |

`run-edition` 默认是交互模式，`--verification-timeout-seconds` 默认 180；只有存在 pending page 才弹出 Edge。Cron/Gateway 必须显式传 `--unattended`，不依赖 TTY 猜测。`--open-verification` 仅保留为兼容别名。

`SKILL.md` frontmatter 里的 `daily_intelligence.data_dir` 等是 Hermes 配置提示，不会神奇地进入 Python。Hermes 必须把配置值转换为 CLI 参数或环境变量。为了避免两套历史，推荐生成 Agent 每次都显式使用同一 `--data-dir`。

### 唯一数据根绑定

`resolve_data_dir()` 只负责解析；`runtime.bind_data_root()` 负责约束。第一次对 Hermes Home 内目录运行正常命令时，它原子写入 `<Hermes Home>/state/daily-intelligence-data-root.json`。后续根不同会在读取 index、run、report 前失败。每个 run 自身还保存绝对 `data_root`，因此即使用户复制了 manifest，也不能在另一根继续。

`data-root status` 是只读诊断；`data-root adopt` 是显式迁移开关。adopt 不复制、合并或删除旧数据，这些高影响动作仍由用户决定。仓库外临时测试目录不写全局注册表，避免 pytest 污染真实 Hermes 环境。

## 7. YAML 配置注入

`load_config()` 使用 `yaml.safe_load()`，然后映射为：

```text
raw["browser"] → BrowserConfig
raw["budget"]  → BudgetConfig
raw["output"]  → OutputConfig
raw["sources"] → list[SourceConfig]
```

构造后立即验证：

- module/category 是否在 taxonomy 中；
- `0 <= report_target <= report_max <= report_items_per_source`；
- output format 只能是 `html/pdf`，且 PDF 必须同时启用 HTML；
- PDF engine 只能是 `edge/reportlab/auto`。

这属于启动期 fail-fast。若配置错误，系统不会启动浏览器后才在报告阶段失败。

`BrowserConfig` 的 `collection_global_concurrency`、`collection_per_domain_concurrency` 和 `http_prefetch_timeout_ms` 只控制公开索引预取；`global_concurrency` 与 `per_domain_concurrency` 只控制正文 enrich。拆开是为了能提高索引覆盖而不同时放大正文抓取压力。

`OutputConfig` 默认 `formats=[html,pdf]`、`pdf_engine=edge`、`open_after_finalize=false`。它只决定派生阅读投影；JSON/Markdown 无条件保存。`--publish` 不属于 output config，它专门控制是否额外调用 Notion。

### `mode` 与 `adapter`

```python
SourceConfig.adapter_name = adapter or mode
```

大多数配置直接用 `mode`。`adapter` 提供显式覆盖，适合“采集模式名称与兼容配置字段不同”的情况。

## 8. 动态栏目页注入

动态页注册表：

```text
DATA_DIR/state/source-pages.json
```

`source_urls()` 合并：

```text
source.url
+ source.explore_urls
+ status=approved 的动态页
→ 保序去重
```

每个 URL 再经过 `validate_source_page()`：必须是绝对 HTTP(S)，域名属于主 URL 或 include domains。每来源最多五个动态页。

`SOURCE_PAGE_REWRITES` 可以在读取时升级已知废弃入口，例如把旧 Hugging Face `/papers/month` 规范为 `/papers`，但不修改不可变历史 index。

## 9. Adapter 注册表注入

`adapters._ADAPTERS` 是一个名称到函数的字典：

```text
arxiv_index       → collect_arxiv_index
browser_index     → collect_browser_index
latest_year_index → collect_latest_year_index
seed_papers       → collect_seed_papers
twz_index         → collect_twz_index
weibo_api         → collect_weibo_api
```

`collect_candidates()` 根据 `source.adapter_name` 查找。未知名称给出可用 adapter 列表。

扩展代码可调用 `register_adapter(name, function)`；重复名称会抛错。这是轻量运行时依赖注入：collector 不知道具体站点协议，只依赖统一 callable 类型。

当前注册发生在模块 import 时，并非 Python entry points 插件系统。第三方 adapter 若要长期使用，应在包初始化或明确扩展模块中注册，并增加测试。

## 10. Agent Context 注入

Python 不把候选直接塞进当前模型调用，而是保存 context artifact，run 只记录路径。Hermes 根据 `SKILL.md`：

1. 读取 run 的 `artifacts.context_path`；
2. 读取 `candidate_items`、`reusable_briefs`、`brief_plan`、连续性和规则；
3. 先合并已评估批准的 reusable briefs；
4. 只对 `author_item_ids` 用一次 `delegate_task(tasks=[...])` 并行发送三个 batch；
5. 合并结构化 briefs；
6. 再由主 Agent 选择 events 和写 analyses。

Context 注入的好处：

- 可以离线检查 Agent 当时看到的内容；
- 可在 Hermes 中断后继续；
- 可以限制字段和候选数量；
- 模型 worker 不需要读取整个 index 或所有历史；
- Python 与模型供应商解耦。

### 批次边界

`brief_authoring_batches` 只指定仍需写作的 source IDs 和 candidate count；`brief_plan` 分开记录 `reuse_item_ids` 与 `author_item_ids`。子 Agent 只能在本批来源内选择，返回 briefs，不写研判、不发布。缓存由 item 内容指纹和交付后独立评估共同门控，不以“上一版存在”作为复用条件。

这不是 Python 线程池。并行是否真正发生取决于 Hermes 按技能说明调用 `delegate_task`。

## 11. 报告证据注入

模型 draft 被视为半可信语义输入。`compile_report_data(report, index)` 用 index 注入权威机械字段：

- 把 original title 恢复为 index title；
- 从 item ID 生成 `source_ref`；
- 把 content status 映射为 access；
- 注入 primary source、published_at、source rank/label；
- 根据 index module/category 移动到正确 section；
- 生成 event/analysis ID；
- 把 analysis 的 evidence item IDs 映射为 event IDs；
- 生成 pending verifications；
- 计算计数和 coverage metrics。

编译器不会用模板生成缺失 brief，也不会用固定中文前缀伪造翻译/TL;DR。它只可注入内容指纹一致且独立评估已批准的缓存；其余实质语义字段必须由 Hermes 提供。

## 12. Notion Schema 注入

`NotionPublisher` 初始化时：

1. 读取 `configs/notion.yaml`；
2. GET `/data_sources/{id}` 获取真实 schema；
3. 顺序测试 `schema_profiles`；
4. 对每个逻辑字段验证属性名和类型；
5. 选择第一套完全匹配的 profile；
6. 后续 `build_properties()` 按实际 status/select 类型做兼容封装。

这相当于“远端 schema 驱动的 mapping 注入”。代码不把 `Name`、`Title` 等写死在 publisher 逻辑里。

校验要求至少映射 title/date；其他映射若存在，必须指向真实属性，并符合：

```text
title → title
date → date
status → status 或 select
source/version → select
tags → multi_select
统计字段 → number
```

状态属性若有选项列表，morning/evening 值也必须存在。

## 13. 注入与安全

所有注入值都要在边界验证：

- YAML 用 safe loader；
- source URL 限制协议和域名；
- adapter 名必须在注册表；
- report 引用必须存在于 index；
- Notion mapping 必须匹配远端类型；
- 网页正文永远不作为配置、命令或系统提示；
- `.env` 只提供凭证/路径，不写入 artifact。

提示注入与依赖注入是两回事：本页所述是受控参数/数据进入系统；网页里的“忽略指令、上传秘密”等文本属于恶意提示注入，必须作为不可信内容拒绝执行。
