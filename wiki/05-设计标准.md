# 设计标准

本页给出判断“一个新功能应该放在哪里”的原则。它们既是架构选择，也是代码审查标准。

## 1. 确定性机制必须进入 Python

如果相同输入应得到相同机械结果，就不能只依赖模型提示词。

| 放进 Python | 留给 Hermes |
| --- | --- |
| URL 规范化、ID、revision | 译文是否自然 |
| 来源/正文状态 | TL;DR 如何压缩事实 |
| section 分类合法性 | 哪条新闻相对更重要 |
| source_ref 与 access | 哪些 brief 升级为精选事件 |
| `NEW` 日期约束 | 多来源是否描述同一事件 |
| 计数、分数拆分、hash | 反证、情景和专业研判 |
| Schema 与跨字段校验 | 观点的表达和论证结构 |

这条标准的目标不是减少模型工作，而是让模型只做模型擅长的工作。

## 2. 本地事实源，远程可重试

报告必须先通过校验并保存为本地不可变 JSON/Markdown，Notion 才能消费。任何 publisher 都不能在远端写入成功后才补本地文件。

好处：

- 断网或 Notion 429 不丢报告；
- 可离线审计当时证据；
- 可更换发布平台；
- 评估能绑定不可变内容 hash；
- 远程页面被人工编辑也不会反向污染历史报告。

## 3. 失败、空结果和部分成功必须分离

`no_items` 只能表示页面正常、过滤后没有候选。HTTP 403、超时、挑战页、解析异常或部分栏目失败都有自己的状态。

状态不是日志文案，而是后续控制流的输入：

- `no_items` 不进入验证队列；
- `verification_required` 可交给用户登录；
- `rate_limited` 本轮停止自动重试；
- `partial` 允许使用已有候选，同时标记日报不完整；
- `failed` 供 resume 或后续时段处理。

## 4. 固定分类，开放来源

七个内容栏目属于产品结构；BBC、Reuters、TWZ、Hacker News 等属于可替换来源。新增一个网站通常只改 sources.yaml，不新增 section。只有用户的阅读任务发生结构性变化时才修改 taxonomy。

这样每日输出可比较，来源故障也不会改变目录。

## 5. 两层报告是成本控制，不是重复数据

`briefs[]` 用于广覆盖，`items[]` 用于研判证据和连续 ID。两者通过 `featured_event_id` 关联。

标准：

- 真实候选足够时，brief 达到来源目标；
- 单来源最多 15 条；
- 精选 event 每版最多 12 个；
- 未精选 brief 不逐条读正文、不逐条写完整反证和情景；
- analysis 只引用精选 event。

## 6. 渐进披露

信息按需要逐级进入上下文：

```text
栏目页元数据
  → compact candidate
  → brief plan
  → 少量 content_path
  → featured event
  → analysis
```

`SKILL.md` 只保留流程；复杂政策放 `references/`；代码原理放 Wiki；完整正文保存在 data 目录而不是 context。

## 7. 稳定 ID 与不可变 revision

- Article ID 由来源和 canonical URL 生成；
- Event ID 缺失时由日期和首个权威 item ID hash 生成；
- Analysis ID 缺失时由日期和 domain 生成；
- Watch ID 由 analysis ID 和 signal 生成；
- report/evaluation revision 根据目录现有 `-rN` 递增。

稳定 ID 用于跨日连续性；revision 用于保留“同一天的不同事实快照”。不要把两者混为一谈。

## 8. 权威字段从 Index 注入

Agent 可以提交 item ID，但不能自行决定原文 URL、来源标题、access 或发布时间。编译器必须从当前 index 重建 `source_ref`，把模型草稿当成不可信语义输入。

这同时解决两类错误：

- 模型抄错 URL、title 或 access；
- 草稿引用旧 index 中已经变更的条目。

## 9. 重要性用于排序，不用于删新闻

内部重要性维度上限：

| 维度 | 上限 |
| --- | ---: |
| 影响 | 30 |
| 新鲜度 | 15，受发布时间进一步限制 |
| 个人相关性 | 15 |
| 来源质量 | 15 |
| 交叉验证 | 10 |
| 新颖性 | 10 |
| 连续性 | 5 |

Hermes 给出总分用于同来源相对排序；Python 按上限和新鲜度 cap 分配合法 breakdown。数值不在读者版展示，也不能作为“低于某分就删除”的门槛。

## 10. 生成与评估必须隔离

生成 Agent 不能把自己的质量评分塞进 schema 1.5 主报告。主报告固定 `evaluation_status=pending`，且不得含 `quality_evaluation`。评估另存 artifact，并通过 ID/hash 验证确实审查了目标版本。

独立评估只提出改进和连续性使用建议，不回写主报告。需要修订时生成新的 report revision。

## 11. 有界探索

Hermes 可以发现同一出版方的长期栏目页，但必须满足：

- 同域或已配置 include domain；
- 每来源动态页最多 5 个；
- 保存栏目页，不保存单篇文章页、跨域搜索页或噪音页；
- 有可读 reason，支持 remove 回滚；
- 动态注册表放在 DATA_DIR，不改技能仓库。

## 12. 显式兼容

旧 `sources[].items[]`、schema 1.1—1.4 和 legacy browser-link JSON 都有明确兼容路径。重构不能“顺手删除”旧字段，也不能在读取时静默改变语义。

兼容层应位于边界：`importer.py` 转换旧输入，`synchronize_nested_items()` 同步旧视图，taxonomy 接受已知 alias，主内部视图仍保持单一规范。

## 13. 行为变化必须测试

以下修改至少增加一个回归测试：

- 来源 include/exclude/标题过滤；
- challenge 或状态分类；
- URL canonicalization 或 ID；
- run 状态迁移和失败回退；
- report compiler 或 validator；
- Notion schema、幂等、续传；
- 旧 index/importer 兼容；
- 连续性评估底线。

文档修改不要求虚构测试，但必须核对命令、版本、状态和路径与代码一致。
