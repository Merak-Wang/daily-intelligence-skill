# 端到端流程

本页按一次真实运行的控制流说明每阶段输入、调用模块、输出文件、状态和恢复方式。所有 CLI 全局参数都必须放在子命令之前。

```powershell
$DataDir = "$env:LOCALAPPDATA\hermes\data\daily-intelligence"
$Profile = "$env:LOCALAPPDATA\hermes\browser-profiles\daily-intelligence"
```

## 0. CLI 启动

每个命令都先经过 `cli.main()`：

```text
argparse
  → load_hermes_environment()
  → load_config(--config, --timezone)
  → resolve_data_dir(--data-dir)
  → 按 subcommand 分发
```

`load_hermes_environment()` 读取 Hermes Home 下的 `.env`，`override=False`，因此当前进程已经设置的变量优先。

## 1. Prepare：采集并生成上下文

```powershell
daily-intel --data-dir $DataDir run-edition `
  --edition morning --profile-dir $Profile `
  --open-verification --verification-timeout-seconds 180
```

以上是交互式 Hermes Desktop 推荐命令。`--open-verification` 会在采集和初始 context 完成后检查 index：存在 `failed`、`verification_required` 或 `rate_limited` 页面时，自动进入第 7 节的已连接 Edge 队列；没有待处理页面则直接返回。Cron/Gateway 无人值守命令必须省略该参数。

调用链：

```text
cli.main
└─ workflow.prepare_edition
   ├─ storage.exclusive_lock
   ├─ notion.sync_user_feedback        失败只记 warning
   ├─ collector.collect_sources
   │  ├─ config.source_urls
   │  ├─ collector.collect_source
   │  └─ adapters.collect_candidates
   ├─ collector.write_results_index
   └─ context.build_context
```

### 1.1 创建 run

路径：`runs/YYYY-MM-DD/<edition>.json`。

run 包含 collection window、budget、deadline、attempt、artifacts、pending_sources、阶段时间戳和 next_action。相同日期/edition 若已有非终态 run，默认直接返回，不重复采集；已完成 run 也直接返回。只有 `--restart` 才开始新 attempt。

### 1.2 同步人工反馈

如果配置了 Notion，prepare 会查找页面末尾形如：

```text
用户反馈|相关性=|准确性=|分析价值=|整体满意度=|补充意见=
```

解析后的反馈写入 `state/user-feedback.json`。远程读取失败只进入 `feedback_sync_warning`，不阻塞采集。

### 1.3 采集来源

Windows 默认启动持久化 Edge context；无人值守时 `headless=True`。当前实现按 sources.yaml 顺序采集来源，每个来源内部按主 URL、`explore_urls`、动态栏目页顺序访问。

每个页面：

1. `goto(..., wait_until="domcontentloaded")`；
2. 等待来源 `wait_ms` 或默认 3500ms；
3. 检测 HTTP、挑战文本、验证码 iframe 和限流文本；
4. 无挑战时调用 adapter；
5. 过滤并规范化候选；
6. 返回页面级 `SourceResult`。

同来源多个页面通过 round-robin 合并，避免 BBC 首页 80 条把 Business/Science 栏目全部挤掉。去重后重新写入来源级 `source_rank`。

输出：`indexes/YYYY-MM-DD/<edition>-rN.json`，并更新 `indexes/latest.json`。

### 1.4 构建 context

`context.build_context()` 压缩候选、读取最近报告和连续状态，生成三批 `brief_authoring_batches` 与逐来源 `brief_plan`。

输出：`context/YYYY-MM-DD/<edition>-rN.json`，run 进入 `awaiting_selection`。

## 2. Hermes 编写覆盖层并选择正文

Python 到此不会自动翻译，也不会自动生成 TL;DR。Hermes 读取 `artifacts.context_path` 后：

1. 检查 `brief_plan` 非空；
2. 一次调用 `delegate_task(tasks=[...])`，把三个 batch 同时交给三个子 Agent；
3. 每个子 Agent 只返回本批 briefs，不做研判、不发布；
4. 主生成 Agent 合并批次并按相对重要性排序；
5. 从 briefs 中选择通常 6—10、最多 12 个精选事件候选；
6. 把需要正文的 item ID 按重要性顺序传给 enrich。

`default_item_ids` 是 Python 给出的确定性基线，可以换成同来源其他候选，但不能跨来源补数量。

## 3. Enrich：按需读取精选正文

```powershell
daily-intel --data-dir $DataDir enrich-edition `
  --run "$DataDir\runs\2026-07-17\morning.json" `
  --item-id ITEM_A --item-id ITEM_B `
  --profile-dir $Profile
```

调用链：

```text
cli.main
└─ workflow.enrich_edition
   ├─ 读取 previous selected IDs 与剩余硬预算
   ├─ content.extract_content
   │  ├─ _ordered_targets
   │  ├─ _run_parallel_extraction
   │  ├─ _extract_one
   │  └─ synchronize_nested_items
   └─ context.build_context
```

规则：

- 保留调用者传入的重要性顺序；
- 去重并忽略未知 ID；
- 每版累计不超过 `max_fulltext_per_run=12`；
- deadline 已过时不再新增正文；
- 全局 semaphore 默认 3，同域 semaphore 默认 1；
- 先取同域槽，再取全局槽，防止同域等待者占满全局并发。

正文清除 script、style、nav、footer、广告、cookie 等噪音节点，再按 `content_selectors` 找第一个至少 500 字符的区域。正文写入 `content/<source>/<item>/<timestamp>.md`，index 生成新 revision，context 也生成新 revision。

如果不需要正文，也要执行：

```powershell
daily-intel --data-dir $DataDir enrich-edition `
  --run "$DataDir\runs\2026-07-17\morning.json" --max-items 0
```

这不会调用 `content.extract_content()`，但会刷新 context 并把 run 推进到 `awaiting_authoring`。

## 4. Hermes 完成精选事件和研判

主生成 Agent 使用最新 context：

- 为外文 brief 保留原始 `title`，另写自然中文 `title_zh`；
- TL;DR 证据优先级为正文 > 公开 description/abstract > 标题明确事实；
- 不写“详见原文”“暂未获取中文摘要”等占位语；
- 精选 events 引用 exact item ID；
- 三个 analysis domain 分开编写；
- 地缘政治覆盖中国与西方立场、历史脉络和辩证关系；
- AI 视角覆盖技术趋势、工程意义、学习/研究顺序；
- 市场视角覆盖趋势、风险、反证和观察信号；
- 晚报必须写 `changes[]` 和 `tomorrow_watch_items[]`。

输出是语义 draft，不必手写 event ID、source_ref、计数或分项分数，这些交给编译器。

## 5. Finalize：编译、校验、保存、发布

```powershell
daily-intel --data-dir $DataDir finalize-edition `
  --run "$DataDir\runs\2026-07-17\morning.json" `
  --report "C:\path\to\draft.json" --publish
```

调用链：

```text
workflow.finalize_edition
├─ reports.save_report
│  ├─ reporting.compile_report_data
│  ├─ reporting.normalize_report_data
│  ├─ reporting.validate_report_data
│  ├─ storage.write_immutable_json
│  └─ reports.render_report_markdown
├─ notion.publish_report                 --publish 时
│  ├─ 再次 validate_report
│  ├─ Notion schema profile 匹配
│  ├─ 创建/复用日期页面
│  └─ 每 100 blocks 追加并记录进度
└─ schedule_independent_evaluation       发布成功后
```

### 5.1 编译失败

`save_report()` 抛错时，workflow 把 run 恢复为 `awaiting_authoring`，记录可执行错误和 `next_action`。修正 draft 后重跑 finalize，不需要重新采集或 enrich。

### 5.2 保存成功

输出：

```text
reports/YYYY-MM-DD/<edition>-rN.json
reports/YYYY-MM-DD/<edition>-rN.md
reports/latest-<edition>.json
```

JSON revision 不可覆盖；Markdown 使用临时文件再原子替换目标。run 记录 report_id、路径、warnings、hash 和 state_paths。

### 5.3 Notion 失败

本地报告已经有效。run 保持 `publishing` 并记录 error；修复网络、token 或 schema 后重新运行同一 finalize。`notion-registry.json` 保存已追加 block 数，避免从头重复整页。

## 6. 独立评估

成功发布后，workflow 运行：

```text
hermes cron create 2m <绑定报告路径、索引、report_id、hash 的提示>
  --repeat 3
  --skill daily-intelligence
  --deliver local
```

生成日报的当前命令立即返回。独立任务读取不可变报告和契约，写 evaluation draft，然后调用：

```powershell
daily-intel --data-dir $DataDir finalize-evaluation `
  --report "...\morning-r1.json" `
  --evaluation "...\evaluation-draft.json" --publish
```

`save_evaluation()` 校验 report ID/hash，保存不可变评估，按质量底线更新连续状态；`append_evaluation()` 把评估追加到同一 Notion 页面。调度失败只写入 run.evaluation.scheduler，不撤回日报；再次对已完成 run 执行 `finalize-edition --publish` 可尝试重新调度。

## 7. Edge 人工验证与补充 revision

若 `run-edition` 没有使用自动打开参数，或需要稍后重开：

```powershell
daily-intel --data-dir $DataDir verify-pending `
  --index "...\morning-r1.json" `
  --browser-channel msedge --profile-dir $Profile --timeout-seconds 90
```

流程：

1. `run-edition --open-verification` 或 `verify-pending` 调用同一执行函数，从 index 聚合 `failed`、`verification_required`、`rate_limited` 页面；
2. 生成本地验证队列 HTML，并由 CLI 启动本地桥接；
3. 用户点击来源链接，在同一个持久化 Edge context 中打开；
4. CLI 轮询页面，验证通过且出现合格条目时立即调用 `collect_loaded_page()`；
5. 成功结果与原 index 合并为新 revision；
6. `adopt_index_for_run()` 重建 context。

403、关闭、超时、无条目或临时限流只更新 UI 状态并保留链接，不阻塞其他来源。若原 run 已完成，系统保存 `previous_report`，清除旧 publication/evaluation 引用，把 run 置回 `awaiting_selection`，下一版成为可追溯补充 revision。

## 8. 恢复矩阵

| run 状态 | 下一步 |
| --- | --- |
| `awaiting_selection` | 选择 item ID，运行 enrich；或 `--max-items 0` |
| `extracting_content` | 检查浏览器是否中断；确认锁后重新运行 enrich |
| `awaiting_authoring` | 修正/完成 draft，运行 finalize |
| `finalizing` | 可重跑 finalize；保存逻辑会检查已有 artifact |
| `publishing` | 修复 Notion 后重跑 finalize `--publish` |
| `completed_partial` | 报告可用；按需运行 verify-pending 并生成补充 revision |
| `failed` | 查看 error；`run-edition --restart` 开始新 attempt |

删除 lock 前必须确认没有真实进程仍在运行。lock 文件没有自动 stale 判定，误删会造成并发写入。
