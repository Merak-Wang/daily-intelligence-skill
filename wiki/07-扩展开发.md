# 扩展开发

## 1. 先判断扩展类型

| 需求 | 最小修改范围 |
| --- | --- |
| 同一来源新增栏目页 | `source-page add` 或 `explore_urls` |
| 新增普通 HTML 新闻站 | `configs/sources.yaml` + 过滤测试 |
| 新增特殊 DOM/API 协议 | `adapters.py` + config + 单元测试 |
| 新增产品栏目 | taxonomy + schema + compiler + renderer + contract + tests + docs |
| 新增报告字段 | schema + compiler/validator + renderer + state/publisher + tests |
| 新增发布平台 | 独立 publisher，只消费已保存 report |

优先使用最小范围，不为一个站点新建运行脚本。

## 2. 添加普通网页来源

```yaml
- id: example_news
  name: Example News
  enabled: true
  mode: browser_index
  url: https://example.com/news
  explore_urls:
    - https://example.com/business
  include_domains:
    - example.com
  max_items: 60
  report_target: 6
  report_max: 15
  role: evidence
  article_patterns:
    - '^https://example\.com/(news|business)/.+$'
  exclude_patterns:
    - '/author/'
    - '/privacy/'
    - '/video/'
  exclude_title_patterns:
    - '^view all comments$'
  content_selectors:
    - article
    - main article
    - main
  module: information
  category: international
  language: en
  region: global
```

### 字段设计顺序

1. 先确定 `module/category`，它必须通过 taxonomy；
2. 再确定 role：聚合/热榜通常是 discovery，官方发布通常是 primary，专业媒体通常是 evidence；
3. 限制 include domain，防止广告和跨站跳转；
4. article pattern 尽可能只匹配文章详情页；
5. exclude 掉作者、标签、视频、隐私、登录、评论等；
6. 根据真实产出调整 `max_items` 与 `report_target`，不是越大越好；
7. 写至少一个有效链接和一个噪音链接测试。

配置加载时要求：

```text
0 <= report_target <= report_max <= budget.report_items_per_source
```

当前全局上限为 15。

## 3. 添加静态或动态栏目页

稳定、对所有安装都有价值的栏目写入 `explore_urls`。只对当前用户有价值、仍在试验的栏目用：

```powershell
daily-intel --data-dir DATA_DIR source-page add `
  --source example_news --url https://example.com/science `
  --reason "补充长期科学栏目"
```

动态页写入 `DATA_DIR/state/source-pages.json`，每来源最多 5 个。`validate_source_page()` 要求绝对 HTTP(S) URL 且域名属于 `include_domains` 或主 URL 域名。

不要加入：

- 单篇文章；
- 站内搜索结果；
- 登录/账户页面；
- 跨域聚合页面；
- 只在一次事件中临时有用的页面。

## 4. 添加 adapter

默认 `browser_index` 会遍历页面所有 anchor，再用配置过滤。以下情况才需要新 adapter：

- 页面通过 JSON API 返回结构化数据；
- 标题和链接不在普通 anchor 中；
- 发布时间需要特殊 DOM 关系；
- 需要先跳到“最新年份”二级索引；
- 站点存在稳定且明显更可靠的接口。

接口类型：

```python
IndexAdapter = Callable[[Page, SourceConfig, str], list[ArticleItem]]
```

示例：

```python
def collect_example_api(page, source, discovered_at):
    payload = page.locator("body").inner_text()
    rows = json.loads(payload)["items"]
    return [
        _article(source, row["title"], row["url"], discovered_at)
        for row in rows
    ]

register_adapter("example_api", collect_example_api)
```

然后在 YAML 使用 `mode: example_api`，或 `adapter: example_api`。`adapter` 字段存在时优先于 mode。

Adapter 必须：

- 输出统一 ArticleItem；
- 使用 `_article()` 获得 canonical URL 和稳定 ID；
- 尊重 `max_items`；
- 尽可能填公开 description、published_at 和来源原始排名；
- 不写 run、index、report 或 Notion；
- 不吞掉异常并伪造空列表。

## 5. 为什么不改 collector

Collector 负责所有来源共有的机制：浏览器、挑战、页面级状态、多栏目合并、revision。Adapter 负责来源协议。如果在 collector 里按 source ID 加 `if/elif`，会让一次站点 DOM 变化影响核心生命周期，并让测试组合爆炸。

## 6. 添加分类

分类是产品级破坏性变更，至少同步：

1. `taxonomy.py` 的 enum、required sections、顺序和标题；
2. `schemas/report.schema.json`；
3. `templates/report-contract.md`；
4. `reporting.py` section 编译与验证；
5. `reports.py` Markdown 分组；
6. `notion.py` block 分组；
7. `configs/sources.yaml`；
8. 旧 schema alias/兼容策略；
9. architecture/reporting tests；
10. README 与 Wiki。

不要为“新增一个网站”新增分类。

## 7. 添加报告字段

先回答：

- 字段属于 Brief、Event、Analysis、Run、Evaluation 还是 Publisher？
- 是模型语义还是 Python 可确定值？
- 是否需要跨日保存？
- 数据真源是 index、正文、模型还是用户反馈？
- 缺失时是 error、warning 还是合法 optional？
- 旧报告如何读取？

典型跨模块路径：

```text
schema
→ report-contract
→ compile_report_data / validate_report_data
→ Markdown renderer
→ Notion renderer
→ state（若连续）
→ tests
→ docs
```

## 8. 新增 publisher

新 publisher 应：

- 只读取已保存 report JSON；
- 发布前再次执行 `validate_report()`；
- 有自己的幂等键和断点记录；
- 不修改 report；
- 失败不删除本地 artifact；
- 不读取网页、content 或 Hermes context；
- 有 mock/fixture 测试，不依赖真实凭证跑单元测试。

## 9. 旧格式兼容

`importer.py` 支持旧 browser-link JSON 列表，并把已知来源名称映射到 source ID。未知来源使用规范化名称生成临时 ID。

修改兼容代码时必须保留：

- HTTP 429 → `rate_limited`；
- 401/403 → `verification_required`；
- 根级 `items[]` 规范视图；
- nested items 兼容；
- 无效 status → `failed`，而不是 `no_items`。
